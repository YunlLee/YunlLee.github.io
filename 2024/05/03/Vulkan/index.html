2.2 给实例添加验证层

验证层（Validation Layers）是用于帮助开发者在开发和调试阶段捕捉错误和潜在问题的工具。验证层会在 Vulkan API 调用之前和之后插入额外的代码，以检查应用程序的行为是否符合 Vulkan 规范，并报告任何违规行为。

3. 挑选一个物理设备

所谓得物理设备，就是指CPU。它是不可以创建和销毁，我们只能根据主机得配置挑选合适得设备。

  • Instance::enumeratePhysicalDevices( Dispatch const & d )
    • 枚举所有得物理设备，并返回一个 std::vector<vk::PhysicalDevice>
  • getProperties()
    • 这是 PhysicalDevice 类中的一个函数
    • 返回的是 PhysicalDeviceProperties，包含了deviceType、deviceName等字段。
    • deviceType 的类型是 PhysicalDeviceType，这是一个枚举类
  • enumerateDeviceExtensionProperties()
    • 这也是 PhysicalDevice 的函数
    • 返回的是一个 ExtensionProperties 的集合，它保存了这个设备支持的所有扩展
      • 这个结构体有两个字段：extensionName, specVersion

7. 交换链 SwapChain

创建交换链需要三个扩展，VK_KHR_surface, VK_KHR_win32_surface 和 VK_KHR_swapchain 前两个扩展属于 instance 层级， 最后一个属于 device 层级。

  • 创建swapChain需要使用的结构体： vk::SwapchainCreateInfoKHR createInfo{};

解析 SwapchainCreateInfoKHR 结构体：

  • imageFormat 和 imageColorSpce：像素格式和颜色空间 getSurfaceFormatesKHR 可以获得设备支持的 format 和 colorspace
  • imageExtent：指定了交换链图像的分辨率，即每个交换链图像的宽度(width)和高度(height)。这决定了渲染到交换链图像时所使用的像素数量。
  • imageArrayLayers：是多视图或立体图中表面(surface)的数量，对于非立体的应用这个值是 1。
  • imageUsage：用于指定图像的用途。对于交换链来讲此处为 vk::ImageUsageFlagBits::eColorAttachment。颜色附件通常与渲染通道（Render Pass）和帧缓冲区（Framebuffer）关联。在创建渲染通道和帧缓冲区时，需要指定使用的颜色附件.
  • imageSharingMode：
    • eConcurrent：支持从多个队列族中获取对象的范围或图像的子资源
    • eExclusive：只支持从单个队列中获取对象的范围或图像的子资源
  • compositeAlpha：指定了在复合（compositing）过程中如何处理图像的 alpha 分量。在窗口系统或显示器上呈现图像时，通常会将多个图像层进行复合。这意味着将它们合并在一起形成最终的显示图像。
    • eOpaque：在图片复合期间 alpha 值会被忽略，相当于 alpha = 1；
    • ePreMultiplied：在预乘 Alpha 模式下，复合过程中会考虑图像的 alpha 分量。复合器会正确处理这些预乘过的颜色分量，而不会再次乘以 alpha。图像的颜色分量（如红色、绿色、蓝色）已经被乘以 alpha 分量。也就是说，存储在图像中的颜色值已经是 color * alpha 的结果。
    • ePostMultiplied：表示图像的 alpha 分量在复合过程中被尊重，但应用程序不需要预先将图像的非 alpha 分量乘以 alpha 分量。相反，复合器（compositor）将在复合过程中自行将各个图像非 alpha 分量乘以 alpha 分量。
    • eInherit：交换链不会指定特定的 alpha 复合方式，而是使用窗口系统或显示服务器的默认设置。这意味着应用程序不需要明确指定 alpha 复合行为，而是依赖底层系统的配置。
  • vk::SurfaceFormatKHR 有两个字段：配置 format 和 colorSpace
  • vk::SurfaceCapabilitiesKHR 包含一些 surface 的基本信息
    • 物理设备层函数：
      • getSurfaceCapabilitiesKHR(surface); // 查询 surface 的基本功能
      • getSurfaceFormatsKHR(surface); // 设置 查询surface支持的颜色空间和交换链格式
      • getSurfacePresentModesKHR(surface); // 查询surface支持的显示模式

显示模式：
VK_PRESENT_MODE_MAILBOX_KHR：指定了呈现引擎等待下一个垂直消隐期来更新当前图像。不会出现屏幕撕裂现象。使用内部单项队列来保存待处理的呈现请求。如果队列在接收到新呈现请求时已满，新请求将替换现有条目，与先前条目关联的任何图像将可供应用程序重用。在每个队列非空的垂直消隐期间，将从队列中移除一个请求并进行处理。
VK_PRESENT_MODE_FIFO_KHR：指定了呈现引擎等待下一个垂直消隐期来更新当前图像。不会出现屏幕撕裂现象。使用内部队列来保存待处理的呈现请求。新请求被附加到队列末尾，而在每个队列非空的垂直空白期间，将从队列的开头移除一个请求并进行处理。

8. Image view

在 vulkan 中，一个完整的图像是一个三元组（image, image view, image memory）。

  • image 指交换链创建时生成的图像资源。这些图像资源用于存储将在屏幕上呈现的最终渲染结果。每个 swapchain 包含多个图像，这些图像在交换链的生命周期内进行循环使用。当创建一个 swapchain 时，会生成一组图像，这些图像将被呈现到屏幕上。具体的图像资源可以通过调用 vkGetSwapchainImagesKHR 获取。
  • image memeory: 存储图像数据的实际内存。在创建 VkImage 后，必须为其分配内存并将其绑定到图像。
  • image view: 用于描述如何访问和解释 image（图像）的数据。它指定了图像的格式、子资源范围（例如 mip levels 和 array layers），以及如何在着色器中查看这些图像。

字段解析：
image：要创建视图的图像对象。
viewType：指定图像视图的类型。
format：图像视图的格式，通常与图像的格式相同。
components：指定如何将图像的颜色通道映射到图像视图的颜色通道。

  • eIdentity: 保持颜色通道不变，即使用图像中对应的通道。
  • eZero: 将颜色通道设为0。比如，将红色通道设为0。
  • eOne: 将颜色通道设为1。比如，将红色通道设为1。
  • eR: 将颜色通道重映射为图像的红色通道。例如，可以将绿色通道重映射为红色通道。
  • eG: 将颜色通道重映射为图像的绿色通道。
  • eB: 将颜色通道重映射为图像的蓝色通道。
  • eA: 将颜色通道重映射为图像的Alpha通道。

subresourceRange：指定图像视图的子资源范围。

  • aspectMask：指定图像的访问方面，可以是以下值或其组合。
    • vk::ImageAspectFlagBits::eColor：颜色通道。
    • vk::ImageAspectFlagBits::eDepth：深度通道。
    • vk::ImageAspectFlagBits::eStencil：模板通道。
    • vk::ImageAspectFlagBits::eMetadata：元数据。
  • baseMipLevel：指定开始的 mip 级别索引。
  • levelCount：指定覆盖的 mip 级别数量。如果要覆盖从 baseMipLevel 开始的所有 mip 级别，可以将其设置为 VK_REMAINING_MIP_LEVELS。
  • baseArrayLayer：指定开始的数组层索引。
  • layerCount：指定覆盖的数组层数量。如果要覆盖从 baseArrayLayer 开始的所有数组层，可以将其设置为 VK_REMAINING_ARRAY_LAYERS。
