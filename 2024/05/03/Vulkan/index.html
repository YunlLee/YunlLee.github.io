2.1 给实例添加扩展

Extension 是跨平台的基础。Extension 可以分为个种类：Instance extension 和 另一种 Device extension。An instance extension is one that generally enhances the entire Vulkan system on a platform. This type of extension is either supplied via a device-independent layer or is simply an extension that is exposed by every device on the system and promoted into an instance. Device extensions are extensions that extend the capabilities of one or more devices in the sytem but aren’t necessarily available on every device.

扩展函数指针

7. 交换链 SwapChain

Vulkan 没有“默认帧缓冲区”的概念，因此需要一个基础设施来连接在我们将其显示在屏幕上之前渲染的缓冲区。这个基础设施称为交换链，必须在 Vulkan 中显式创建。交换链本质上是一个等待呈现到屏幕上的图像队列。我们的应用程序将获取这样一个图像进行绘制，然后将其返回到队列中。队列的具体工作方式以及从队列中呈现图像的条件取决于交换链的设置方式，但交换链的一般目的是将图像的呈现与屏幕的刷新率同步。
创建交换链需要三个扩展，VK_KHR_surface, VK_KHR_win32_surface 和 VK_KHR_swapchain 前两个扩展属于 instance 层级， 最后一个属于 device 层级。

  • 创建swapChain需要使用的结构体： vk::SwapchainCreateInfoKHR createInfo{};

解析 SwapchainCreateInfoKHR 结构体：

  • imageFormat 和 imageColorSpce：像素格式和颜色空间 getSurfaceFormatesKHR 可以获得设备支持的 format 和 colorspace
  • imageExtent：指定了交换链图像的分辨率，即每个交换链图像的宽度(width)和高度(height)。这决定了渲染到交换链图像时所使用的像素数量。
  • imageArrayLayers：是多视图或立体图中表面(surface)的数量，对于非立体的应用这个值是 1。
  • imageUsage：用于指定图像的用途。对于交换链来讲此处为 vk::ImageUsageFlagBits::eColorAttachment。颜色附件通常与渲染通道（Render Pass）和帧缓冲区（Framebuffer）关联。在创建渲染通道和帧缓冲区时，需要指定使用的颜色附件.
  • imageSharingMode：
    • eConcurrent：支持从多个队列族中获取对象的范围或图像的子资源
    • eExclusive：只支持从单个队列中获取对象的范围或图像的子资源
  • compositeAlpha：指定了在复合（compositing）过程中如何处理图像的 alpha 分量。在窗口系统或显示器上呈现图像时，通常会将多个图像层进行复合。这意味着将它们合并在一起形成最终的显示图像。
    • eOpaque：在图片复合期间 alpha 值会被忽略，相当于 alpha = 1；
    • ePreMultiplied：在预乘 Alpha 模式下，复合过程中会考虑图像的 alpha 分量。复合器会正确处理这些预乘过的颜色分量，而不会再次乘以 alpha。图像的颜色分量（如红色、绿色、蓝色）已经被乘以 alpha 分量。也就是说，存储在图像中的颜色值已经是 color * alpha 的结果。
    • ePostMultiplied：表示图像的 alpha 分量在复合过程中被尊重，但应用程序不需要预先将图像的非 alpha 分量乘以 alpha 分量。相反，复合器（compositor）将在复合过程中自行将各个图像非 alpha 分量乘以 alpha 分量。
    • eInherit：交换链不会指定特定的 alpha 复合方式，而是使用窗口系统或显示服务器的默认设置。这意味着应用程序不需要明确指定 alpha 复合行为，而是依赖底层系统的配置。
  • vk::SurfaceFormatKHR 有两个字段：配置 format 和 colorSpace
  • vk::SurfaceCapabilitiesKHR 包含一些 surface 的基本信息
    • 物理设备层函数：
      • getSurfaceCapabilitiesKHR(surface); // 查询 surface 的基本功能
      • getSurfaceFormatsKHR(surface); // 设置 查询surface支持的颜色空间和交换链格式
      • getSurfacePresentModesKHR(surface); // 查询surface支持的显示模式

显示模式：
VK_PRESENT_MODE_MAILBOX_KHR：指定了呈现引擎等待下一个垂直消隐期来更新当前图像。不会出现屏幕撕裂现象。使用内部单项队列来保存待处理的呈现请求。如果队列在接收到新呈现请求时已满，新请求将替换现有条目，与先前条目关联的任何图像将可供应用程序重用。在每个队列非空的垂直消隐期间，将从队列中移除一个请求并进行处理。
VK_PRESENT_MODE_FIFO_KHR：指定了呈现引擎等待下一个垂直消隐期来更新当前图像。不会出现屏幕撕裂现象。使用内部队列来保存待处理的呈现请求。新请求被附加到队列末尾，而在每个队列非空的垂直空白期间，将从队列的开头移除一个请求并进行处理。

8. Image view

在 vulkan 中，一个完整的图像是一个三元组（image, image view, image memory）。

  • image 指交换链创建时生成的图像资源。这些图像资源用于存储将在屏幕上呈现的最终渲染结果。每个 swapchain 包含多个图像，这些图像在交换链的生命周期内进行循环使用。当创建一个 swapchain 时，会生成一组图像，这些图像将被呈现到屏幕上。具体的图像资源可以通过调用 vkGetSwapchainImagesKHR 获取。
  • image memeory: 存储图像数据的实际内存。在创建 VkImage 后，必须为其分配内存并将其绑定到图像。
  • image view: 用于描述如何访问和解释 image（图像）的数据。它指定了图像的格式、子资源范围（例如 mip levels 和 array layers），以及如何在着色器中查看这些图像。

字段解析：
image：要创建视图的图像对象。
viewType：指定图像视图的类型。
format：图像视图的格式，通常与图像的格式相同。
components：指定如何将图像的颜色通道映射到图像视图的颜色通道。

  • eIdentity: 保持颜色通道不变，即使用图像中对应的通道。
  • eZero: 将颜色通道设为0。比如，将红色通道设为0。
  • eOne: 将颜色通道设为1。比如，将红色通道设为1。
  • eR: 将颜色通道重映射为图像的红色通道。例如，可以将绿色通道重映射为红色通道。
  • eG: 将颜色通道重映射为图像的绿色通道。
  • eB: 将颜色通道重映射为图像的蓝色通道。
  • eA: 将颜色通道重映射为图像的Alpha通道。

subresourceRange：指定图像视图的子资源范围。

  • aspectMask：指定图像的访问方面，可以是以下值或其组合。
    • vk::ImageAspectFlagBits::eColor：颜色通道。
    • vk::ImageAspectFlagBits::eDepth：深度通道。
    • vk::ImageAspectFlagBits::eStencil：模板通道。
    • vk::ImageAspectFlagBits::eMetadata：元数据。
  • baseMipLevel：指定开始的 mip 级别索引。
  • levelCount：指定覆盖的 mip 级别数量。如果要覆盖从 baseMipLevel 开始的所有 mip 级别，可以将其设置为 VK_REMAINING_MIP_LEVELS。
  • baseArrayLayer：指定开始的数组层索引。
  • layerCount：指定覆盖的数组层数量。如果要覆盖从 baseArrayLayer 开始的所有数组层，可以将其设置为 VK_REMAINING_ARRAY_LAYERS。

10.3 synchronization

信号量

vk::Semaphore 是 Vulkan 中的一个同步原语，用于在 GPU 操作之间进行同步。信号量（Semaphore）用于确保某些操作在其他操作之前完成，通常用于同步命令队列中的命令。

Fence

vk::Fence 是 Vulkan 中的一种同步原语，用于在主机（CPU）和设备（GPU）之间进行同步。与信号量不同，信号量用于在 GPU 操作之间同步，而围栏（Fence）用于从主机监视 GPU 操作的完成情况。

12. 内存管理

有两种基本的资源类型：buffer 和 image。buffer 是一块简单的线性数据，可以用于几乎任何用途——数据结构、原始数组，甚至图像数据，如果你选择这样使用它们的话。而 image 则是有结构的，具有类型和格式信息，可以是多维的，形成自己的数组，并支持从中读取和写入数据的高级操作。这两种类型的资源都是通过两步创建的：首先创建资源本身，然后为资源分配内存。这么做的原因是允许应用程序自行管理内存。内存管理是复杂的，驱动程序很难始终正确地处理它。对于一个应用程序有效的方法可能对另一个应用程序并不适用。因此，期望应用程序能比驱动程序更好地管理内存。例如，一个使用少量非常大的资源并长期保留它们的应用程序，可能会在其内存分配器中使用一种策略，而另一个不断创建和销毁小资源的应用程序可能会实现另一种策略。

1. Buffer

Buffer 常被用于存储线性结构或者非结构化的数据，这些数据可以有一种格式或者原始字符类型。

创建一个新的 buffer 对象：

  • pCreateInfo 的具体定义：
  • flags：关于 buffer 属性的信息。标志字段中定义的唯一可用位是与稀疏缓冲区相关的，现在可以将其设置为 zero。
  • usage：字段告诉 Vulkan 你将如何使用缓冲区，它是由 VkBufferUsageFlagBits 枚举的多个成员组合而成的位域。枚举的多个成员组合而成的位域。在某些架构上，缓冲区的预期用途可能会影响其创建方式。告诉 vulkan 怎么去使用这个 buffer。它是由 VkBufferUsageFlagBits 枚举的成员联合组成的位字段。
  • size：buffer 的大小，以字节为单位。
  • sharingMode 如何在设备支持的多个命令队列上使用缓冲区。因为 Vulkan 可以并行执行许多操作，所以一些实现需要知道缓冲区本质上是一次被单个命令使用，还是可能被许多命令使用。

  • linearTilingFeatures：描述线性平铺格式的支持特性。
  • optimalTilingFeatures：描述最优平铺格式的支持特性。
  • bufferFeatures：描述缓冲区格式的支持特性。

image的属性支持

会将获得的数据存储到 VkImageFormatProperties 结构体中。

  • format：我们需要查询的 image 是否支持的格式。
  • type：VK_IMAGE_TYPE_1D, VK_IMAGE_TYPE_2D, or VK_IMAGE_TYPE_3D
  • tiling：VK_IMAGE_TILING_LINEAR or VK_IMAGE_TILING_OPTIMAL
  • usage：image 的预期用途在这个参数中指定。
  • flags：应该设置为与创建 image 时相同的值。
  • pImageFormatProperties：如果 vulkan 支持该格式，就会将它写入该结构体。
    • maxMipLevels：字段报告了所请求格式的图像支持的最大 mipmap 级别，以及传递给 vkGetPhysicalDeviceImageFormatProperties() 的其他参数。在大多数情况下，当支持 mipmaps 时，maxMipLevels 将报告 log2(max(extent.x, extent.y, extent.z))；当不支持 mipmaps 时，将报告 1。
    • maxArrayLayers：报告了图像支持的最大数组层数。如果支持数组，这个数字可能会相当高；如果不支持数组，则为 1。主要用于指示图像可以具有的数组层的数量。数组层在图形渲染中非常有用，尤其是在实现立体图像、纹理阵列以及其他需要多个图像层的应用场景中。例如，在3D纹理或立方体贴图中，多个数组层允许同时处理多个图像。
    • sampleCounts：如果图像格式支持多重采样，则支持的采样数通过 sampleCounts 字段报告。sampleCounts 是一个位域，其中每个位表示一个支持的采样数。如果第 n 位被设置，则表示该格式支持2n2^n2n个样本。如果该格式被支持，至少会有一个位被设置。几乎不可能看到支持多重采样但不支持每像素单个样本的格式。
    • maxResourceSize：字段指定了使用该格式的资源可能达到的最大大小（以字节为单位）。这一字段不应与最大面积（maxExtent）混淆，后者报告了在每个维度上可能支持的最大尺寸。例如，如果一个实现报告它支持 16,384 × 16,384 像素 × 2,048 层的图像，并且格式包含每像素 128 位，那么创建一个在每个维度上都达到最大扩展的图像将产生 8 TiB 的图像数据。实现不太可能真正支持创建 8 TiB 的图像。然而，它可能确实支持创建一个 8 × 8 × 2,048 的数组图像或一个 16,384 × 16,384 的非数组图像，两者都能适应更适度的内存占用。

5. 内存分配

6. Host Access to Device Memory

设备内存分为多个区域。纯设备内存只能由设备访问。然而，有些内存区域可供主机和设备访问。主机是运行您的应用程序的处理器，您可以请求 Vulkan 为您提供从可由主机访问的区域分配的内存的指针。这称为映射内存。要将设备内存映射到主机的地址空间，要映射的内存对象必须是从具有 VK_MEMORY_PROPERTY_HOST_VISIBLE_BIT 标志设置的堆属性中分配的。假设是这种情况，映射内存以获取主机可用的指针的操作是调用 vkMapMemory()。

刷新操作是必要的，如果主机已经对映射的内存区域进行了写入，并且需要设备看到这些写入的效果。然而，如果设备对映射的内存区域进行了写入，并且您需要主机看到设备写入的效果，您需要使主机上可能包含旧数据的任何缓存无效。

13.1 Record Command

对命令缓冲区的访问必须进行外部同步，这意味着应用程序有责任确保没有两个线程同时尝试向同一个命令缓冲区中记录命令。然而，以下情况是完全可以接受的：

  • 一个线程可以通过连续调用不同命令缓冲区的命令缓冲区函数来记录多个命令缓冲区中的命令。
  • 两个或更多线程可以参与构建一个命令缓冲区，只要应用程序可以保证它们中的任何两个线程从未同时执行命令缓冲区构建函数。

Vulkan 的一个关键设计原则是支持高效的多线程。为此，重要的是确保应用程序的线程不会因为保护共享资源而阻塞彼此的执行，例如通过使用互斥锁。基于这一原因，最好为每个线程分配一个或多个命令缓冲区，而不是尝试共享一个。此外，由于命令缓冲区是从池中分配的，您可以进一步为每个线程创建一个命令池，使工作线程能够从各自的池中分配命令缓冲区而无需相互干扰。

13.2 回收 CommandBuffer。

从池中分配的命令缓冲区不会被 vkResetCommandPool() 释放，但所有命令缓冲区都会返回到它们的初始状态，就像它们是新分配的一样。vkResetCommandPool() 通常在帧结束时使用，将一批可重用的命令缓冲区返回到它们的池中，而不是单独重置每个命令缓冲区。

命令缓冲区可能消耗的资源量本质上是无界的，因为没有硬性限制单个命令缓冲区中可以放置的命令数量。如果应用程序混合使用非常小和非常大的命令缓冲区，最终所有命令缓冲区可能会变得与最复杂的命令缓冲区一样大。

为了避免这种情况，我们可以设置 VK_COMMAND_BUFFER_RESET_RELEASE_RESOURCES_BIT 或 VK_COMMAND_POOL_RESET_RELEASE_RESOURCES_BIT。

13.3 submission of Commands

Calling vkQueueWaitIdle() or vkDeviceWaitIdle() is really not recommended, as they fully flush any work on the queue or device and are very heavyweight operations. Neither should be called in any performance-critical part of your application. Suitable use cases include just before shutting down the application or when reinitializing application subsystems such as thread management, memory management, and so on, where there is likely to be a substantial pause anyway.
